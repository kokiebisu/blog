> 実際にプロジェクトを作っていくに当たってパッケージには２種類の用途があります

## Development 用

こちらの部類のパッケージはあくまで開発を円滑に進めるために必要なパッケージとなっています。

例

- nodemon
- logger
- テスト用のパッケージ

### インストール方法

```bash
npm install -D <パッケージ>
```

```bash
npm install --save-dev <パッケージ>
```

どちらでも可能！

## Production 用

こちらではサービスを実際にネットに公開した際、サービスを機能していくためのパッケージです

- React（フロントエンドの骨格となるもの）
- Express（バックエンドのサーバーの骨格となるもの）

```bash
npm install <パッケージ>
```

# なぜ分ける必要があるのか？

プロジェクトが大きなってくると容量がかなり大きくなってしまいます。（ときには１ギガ以上にも。。。）。ブラウザの性質上、全て読み込まれるまでユーザーはサービスを利用していくことはできません。想像してみてください。1 ギガの容量を読み込むのにどれほどの時間がかかるのことか。この点を踏まえると Production には必要最低限サービスを機能していくために必要なものだけを選んでいく必要があります。

### パッケージのバージョン管理

> パッケージのバージョン管理は Javascript の依存関係を考える上で非常に大事な考え方となってきます

一般的にパッケージは**semantic version notation**を使っています

```jsx
1.0.0
```

## 1(1.)

Major バージョン

コードが根本的に変わった際に(これを breaking changes という）更新されます。

新しい Major バージョンの更新をする際には注意しましょう！今まで使っていた同パッケージが使えなくなる可能性があります

## 2(0.)

Minor バージョン

新しい機能が付け加えられるときに更新されます

## 3(0)

Patch バージョン

ほとんどバグの修正などがされたときに更新されます

## アップデートをする方法

```bash
npm update
```

## ブレーキをかける

バージョンが勝手に更新されないようにストッパーをかけることができます。

```
^1.0.0
```

**^**の記号を前につけることによって、**Minor と Patch のアップデートがあるとき**にだけアップデートします

上記だと：

1.0.15

1.1.23 などは大丈夫ですが

2.0.0 には絶対なりません

```
~1.0.0
```

**~**をつけることによって、**Patch のアップデートがあるときのみ**アップデートします。

~の方がストッパーとしては安全。基本的にはこれを使っていきましょう。

## 更新されなければならないパッケージを確認する方法

```bash
npm outdated
```

## 具体的なバージョンのパッケージをインストールする方法

```bash
npm <モジュール>@<バージョン>
```

### パッケージの消し方

```bash
npm uninstall
```
